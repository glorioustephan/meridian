# Next.js Integration

Meridian integrates with Next.js App Router through a precompile step. Meridian transforms your class source files into standard React TSX before `next dev` or `next build` runs, so Next.js sees only ordinary React code.

## The model

Meridian components are always client components. Every Meridian source file starts with `'use client'`, and the generated output preserves that directive. This aligns with how Next.js App Router works: client components are modules that opt into the browser environment.

Next.js Server Components remain standard async function components. They do not use any Meridian syntax — Meridian's `ServerComponent` base class is deferred to a future release. Server Components can freely import and render Meridian client components as children.

```
Server Component (async function)
└── Meridian client component (generated function with 'use client')
    └── Server Component cannot appear below this boundary
```

This is the standard Next.js client/server boundary model. Meridian adds no special handling — it simply produces valid `'use client'` modules that Next.js understands natively.

## Setup

### 1. Install packages

Follow the [Installation guide](./installation.md) to install `meridian`, `@meridian/compiler`, and `@meridian/cli`.

### 2. Add lifecycle scripts

Add `predev` and `prebuild` scripts to `package.json`:

```json
{
  "scripts": {
    "predev": "meridian build",
    "dev": "next dev",
    "prebuild": "meridian build",
    "build": "next build"
  }
}
```

These `pre*` scripts run automatically before `dev` and `build`. The compiler writes generated files to `.meridian/generated/` before Next.js starts.

### 3. Configure path aliases

Add a path alias in `tsconfig.json` so you can import generated components cleanly:

```json
{
  "compilerOptions": {
    "paths": {
      "@meridian/*": ["./.meridian/generated/*"]
    }
  }
}
```

### 4. Add .gitignore entry

```
.meridian/
```

Generated files are deterministic build artifacts. Do not commit them.

## The precompile workflow

1. `meridian build` scans your `src/` directory for Meridian source files.
2. For each file, it compiles the Meridian classes to React function components and custom hooks.
3. Generated TSX files are written to `.meridian/generated/`, preserving the relative directory structure from `src/`.
4. `next dev` or `next build` starts and processes `.meridian/generated/` as ordinary React TypeScript.

```
src/components/ProductCard.tsx (Meridian source)
    ↓ meridian build
.meridian/generated/components/ProductCard.tsx (generated React)
    ↓ next build
.next/ (Next.js build output)
```

## A complete example

### The project structure

```
my-app/
├── app/
│   ├── layout.tsx
│   └── products/
│       └── page.tsx          ← Server Component
├── src/
│   └── components/
│       └── ProductCard.tsx   ← Meridian source
├── .meridian/
│   └── generated/
│       └── components/
│           └── ProductCard.tsx  ← Generated by meridian build
├── next.config.ts
└── tsconfig.json
```

### The Meridian source component

```tsx
// src/components/ProductCard.tsx
'use client';

import { Component, state, effect } from 'meridian';

interface Product {
  id: string;
  name: string;
  price: number;
  inventory: number;
}

interface ProductCardProps {
  product: Product;
  onAddToCart?: (id: string) => void;
}

export class ProductCard extends Component<ProductCardProps> {
  @state quantity = 1;
  @state addedToCart = false;

  get total(): number {
    return this.props.product.price * this.quantity;
  }

  get canAdd(): boolean {
    return this.quantity <= this.props.product.inventory && !this.addedToCart;
  }

  increment(): void {
    if (this.quantity < this.props.product.inventory) {
      this.quantity = this.quantity + 1;
    }
  }

  decrement(): void {
    if (this.quantity > 1) {
      this.quantity = this.quantity - 1;
    }
  }

  handleAddToCart(): void {
    this.addedToCart = true;
    this.props.onAddToCart?.(this.props.product.id);
  }

  @effect
  resetOnProductChange(): void {
    this.quantity = 1;
    this.addedToCart = false;
  }

  render() {
    return (
      <div className="product-card">
        <h3>{this.props.product.name}</h3>
        <p>${this.props.product.price.toFixed(2)}</p>
        <div className="quantity-selector">
          <button onClick={this.decrement} disabled={this.quantity <= 1}>−</button>
          <span>{this.quantity}</span>
          <button
            onClick={this.increment}
            disabled={this.quantity >= this.props.product.inventory}
          >
            +
          </button>
        </div>
        <p>Total: ${this.total.toFixed(2)}</p>
        <button onClick={this.handleAddToCart} disabled={!this.canAdd}>
          {this.addedToCart ? 'Added!' : 'Add to Cart'}
        </button>
      </div>
    );
  }
}
```

### The Server Component page

```tsx
// app/products/page.tsx (Server Component — no 'use client')
import { ProductCard } from '@meridian/components/ProductCard';

interface Product {
  id: string;
  name: string;
  price: number;
  inventory: number;
}

async function fetchProducts(): Promise<Product[]> {
  const res = await fetch('https://api.example.com/products', {
    next: { revalidate: 60 },
  });
  return res.json();
}

export default async function ProductsPage() {
  const products = await fetchProducts();

  return (
    <main>
      <h1>Products</h1>
      <ul className="product-grid">
        {products.map(product => (
          <li key={product.id}>
            <ProductCard product={product} />
          </li>
        ))}
      </ul>
    </main>
  );
}
```

The Server Component fetches data using `async/await` — standard Next.js App Router patterns. It passes the fetched data as props to `ProductCard`, which is the compiled Meridian client component. Next.js handles the server/client boundary automatically based on the `'use client'` directive in the generated file.

### The generated client component

<details>
<summary>View .meridian/generated/components/ProductCard.tsx</summary>

```tsx
// .meridian/generated/components/ProductCard.tsx
// Generated by Meridian compiler — do not edit manually.
'use client';

import { useState, useEffect } from 'react';

interface Product {
  id: string;
  name: string;
  price: number;
  inventory: number;
}

interface ProductCardProps {
  product: Product;
  onAddToCart?: (id: string) => void;
}

export function ProductCard(props: ProductCardProps) {
  const [quantity, setQuantity] = useState(() => 1);
  const [addedToCart, setAddedToCart] = useState(() => false);

  const total = props.product.price * quantity;
  const canAdd = quantity <= props.product.inventory && !addedToCart;

  function increment() {
    if (quantity < props.product.inventory) {
      setQuantity(quantity + 1);
    }
  }

  function decrement() {
    if (quantity > 1) {
      setQuantity(quantity - 1);
    }
  }

  function handleAddToCart() {
    setAddedToCart(true);
    props.onAddToCart?.(props.product.id);
  }

  useEffect(() => {
    setQuantity(1);
    setAddedToCart(false);
  }, [props.product.id]);

  return (
    <div className="product-card">
      <h3>{props.product.name}</h3>
      <p>${props.product.price.toFixed(2)}</p>
      <div className="quantity-selector">
        <button onClick={decrement} disabled={quantity <= 1}>−</button>
        <span>{quantity}</span>
        <button
          onClick={increment}
          disabled={quantity >= props.product.inventory}
        >
          +
        </button>
      </div>
      <p>Total: ${total.toFixed(2)}</p>
      <button onClick={handleAddToCart} disabled={!canAdd}>
        {addedToCart ? 'Added!' : 'Add to Cart'}
      </button>
    </div>
  );
}
```

</details>

:::tip
The `@effect resetOnProductChange` method reads `this.props.product.id` implicitly — it resets state when the product changes. The compiler infers `[props.product.id]` as the dependency array. No manual tracking required.

Wait — the effect body in the source does *not* read `this.props.product.id`. The effect resets state but reads nothing. In that case, the compiler emits `[]` as the dependency array, meaning it runs once on mount. If you want it to reset when the product changes, you should read `this.props.product.id` somewhere in the effect body, or restructure the component.
:::

## Watch mode for development

During active development, use `meridian watch` so the compiler reacts to file changes immediately:

```bash
# Terminal 1: Meridian watch
pnpm exec meridian watch

# Terminal 2: Next.js dev server
pnpm exec next dev
```

Or use a concurrent script runner:

```json
{
  "scripts": {
    "dev": "concurrently \"meridian watch\" \"next dev\""
  }
}
```

When `meridian watch` detects a source change, it recompiles the affected file and writes the updated generated file. Next.js's dev server picks up the change through its normal HMR mechanism.

## Mixing Meridian and standard React

Meridian components and standard React function components coexist freely in the same Next.js application. There is no requirement to use Meridian everywhere.

```tsx
// A standard React component — no Meridian
export function StaticBadge({ label }: { label: string }) {
  return <span className="badge">{label}</span>;
}

// A Meridian component — compiled from class syntax
export class InteractiveCounter extends Component<{ initial: number }> {
  @state count = this.props.initial;
  increment() { this.count++; }
  render() { return <button onClick={this.increment}>{this.count}</button>; }
}
```

Both can be used side by side in any page or layout. Start with the components where hooks complexity is highest — those are where Meridian provides the most value.

## What Meridian does not do in Next.js

- **Server Components:** Meridian does not provide a `ServerComponent` base class in v1. Keep Server Components as standard async function components.
- **Route handlers:** Route handlers (`app/api/*/route.ts`) are always server-side — Meridian cannot be used there.
- **Middleware:** Middleware runs on the Edge runtime — Meridian components cannot be used in middleware.
- **Server actions:** Use standard React Server Actions. Meridian components can call server actions as regular async functions passed as props.
